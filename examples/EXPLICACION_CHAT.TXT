INSTRUCCIONES PARA IMPLEMENTAR CHAT ASESOR ↔ CLIENTE EN APP FLUTTER

Tenemos una app Flutter con dos entidades separadas:

Usuarios → Asesores

Clientes → Clientes finales

Se necesita implementar un chat 1 a 1 cumpliendo estrictamente estas reglas:

REGLAS DE NEGOCIO (OBLIGATORIAS)

Cada cliente solo puede escribir a su asesor asignado

Un asesor puede escribir a todos sus clientes

No existen chats entre clientes

No existen chats entre asesores

Todas las validaciones se hacen en backend, no en Flutter

ARQUITECTURA REQUERIDA

❌ No crear otro backend

❌ No crear otra base de datos

✅ Integrar el chat como módulo dentro del backend existente

✅ Usar la misma autenticación actual

✅ Chat desacoplado del resto del sistema (usuarios/clientes)

MODELO DE DATOS DEL CHAT
Conversación (1 asesor ↔ 1 cliente)
{
  "_id": "conversationId",
  "asesorId": "usuarioId",
  "clienteId": "clienteId",
  "lastMessageAt": "ISO_DATE"
}

Mensaje
{
  "_id": "messageId",
  "conversationId": "conversationId",
  "senderType": "ASESOR | CLIENTE",
  "senderId": "id",
  "text": "mensaje",
  "createdAt": "ISO_DATE"
}

PERMISOS (BACKEND)

Al enviar o leer mensajes:

Si el usuario autenticado es CLIENTE:

Solo puede acceder a conversaciones donde clienteId = suId

Si el usuario autenticado es ASESOR:

Solo puede acceder a conversaciones donde asesorId = suId

Cualquier otro acceso debe ser rechazado.

ENDPOINTS MÍNIMOS

REST:

GET /chat/conversations

GET /chat/messages/:conversationId

Tiempo real:

WebSocket / Socket.IO

Evento: sendMessage

Evento: receiveMessage

FLUTTER (CONSUMO)

Flutter no decide permisos

Flutter:

Lista conversaciones

Muestra mensajes

Envía mensajes

El backend valida siempre el acceso

OBJETIVO

Implementar un chat:

Simple

Seguro

Escalable

Compatible con la estructura actual de la app

NOTA IMPORTANTE

Este diseño no debe separarse en microservicios ni en otra base de datos.
Si en el futuro el volumen lo requiere, podrá desacoplarse sin rehacer la lógica.
